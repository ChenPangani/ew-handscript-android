## 一、项目初衷与定位

**一句话**: 个人笔迹数字孪生修仙游戏。用户扫描历史手写稿，通过"五行消消乐"式人机协同校对，构建Git版本字库，最终生成手写风格文档。

**创意源起**: 观察朋友使用黑曜石（Obsidian）构建知识库及微信数据沉淀，意识到个人手写数据具有独特的情感价值与数字遗产属性。将企业级ERP方法论个人化（iRP），以"修仙"游戏化包装降低校对枯燥感，以"五行"属性赋予手写体文化叙事。

**目标用户**:
- 种子：学生群体（8-22岁，作业/日记/作文/手稿）
- 付费：成年人（22-60岁，情书/家书/工作笔记/手稿）

## 三、各阶段深度复盘

### 3.1 Phase 1-2：规划与设计（成功范式）

**成功要素**:
- **PRD冻结机制**：产品需求在代码开发前完全锁定，避免开发期需求漂移
- **架构先行**：四层技术架构（PaddleOCR→初框→OpenCV→人机协同）在编码前已定义清晰
- **多Agent并行**：UI/IXD/算法/后端独立推进，通过Owner（人类）做信息路由

**可复用经验**:
- 任何项目启动前，必须完成"PRD+架构书"双冻结
- Agent分工矩阵必须明确"绝对禁止"事项（如UI不碰交互逻辑）

### 3.2 Phase 4-1：导航联调（成功范式）

**成功要素**:
- 单一模块（Navigation）在隔离环境中快速验证
- Compose Navigation + Hilt 依赖注入在最小范围内跑通

### 3.3 Phase 4-2：编译阻塞（失败与重生）

**这是项目最核心的教训区。**

**失败现象**:
- 9轮排查，约15次编译尝试，耗时约8小时（跨两个对话窗口）
- 系统性错误：Agent-D交付的Kotlin代码中`<<`双尖括号泛型错误、`@Serializable`与kapt冲突、Room Entity中`List<String>`字段、`System.currentTimeMillis()`默认值、缺失`GlyphStatistics`类、OpenCV/Timber依赖缺失

**根因分析**:
1. **LLM生成代码的系统性缺陷**：Kotlin泛型语法在LLM输出中高频出现`<<`（如`List<<String>`），这是token级概率错误，非逻辑错误
2. **Room规则盲区**：Agent-D对Room的约束（无List/Map字段、无方法调用默认值）理解不足
3. **依赖缺失后置发现**：Agent-Arch未在首批代码移回前提供完整依赖映射表，导致core/模块编译时才发现OpenCV缺失
4. **Kimi越权修复**：总控窗口在排查中3次直接生成修复代码，违反Wiki规范，造成字段名不一致（`charValue`→`character`）等连锁恶果
5. **跨平台调试不可行**：Kimi无法访问Android Studio的AST、Gradle状态、Build Output实时流，只能依赖Owner截图，信息损耗极大

**止损机制**:
- 编译前系统性扫描脚本（`pre_compile_scan.sh`）
- Agent交付质量门禁（8项强制检查）
- 最大排查轮次限制（≥4轮强制停止）

### 3.4 平台交转实战细节（Cursor/Trae/千问攻坚纪实）

**3.4.1 千问：基础设施诊断（幕后关键突破）**

千问在交转关键节点做了两件 Cursor/Trae 和 Kimi 都未做的事：

1. **诊断 .gitignore 缺失导致 AS 编译暴躁**：此前所有平台（Kimi/Cursor/Trae）都聚焦于代码错误，但千问指出项目缺少标准 `.gitignore`（Android/Kotlin 模板），导致 Gradle 缓存、IDE 索引文件被误提交/污染，引发编译器"暴躁"（无明确错误但构建异常）。这是**根因级诊断**。

2. **设计 Ubuntu 私有 Git 中心方案**：提出利用 `receive.denyCurrentBranch updateInstead` 配置，将 Ubuntu Server 设为私有 Git 中心，实现 iMac/iPad/Mate30 多终端代码同步。这是**长期基础设施**。

3. **提出"AI 全局感知修改"原则**：千问明确建议"禁止手动修改 `build.gradle.kts`，必须通过 AI IDE（Trae）读取全项目上下文后修改，确保 toml/gradle/wrapper 版本一致性"。这成为后续 Cursor/Trae 修复依赖问题的核心方法论。

**3.4.2 Cursor：依赖修复与 API 对齐（首战）**

Cursor 在 IDE 内直接读取 Gradle 错误流，快速解决三类问题：

**A. HiltWorkerFactory 依赖缺失**
- 现象：`HandCraftFontApp` 中注入 `HiltWorkerFactory`，但 Gradle 缺少 `work-runtime-ktx` + `hilt-work` + `hilt-compiler`
- 修复：在 `libs.versions.toml` 和 `app/build.gradle.kts` 中补充三行依赖
- 结果：`:app:kaptDebugKotlin` 通过

**B. OpenCV API 版本陷阱**
- 现象：`OpenCVLoader.initLocal()` 报 `Unresolved reference`
- 根因：Maven 依赖 `com.quickbirdstudios:opencv:4.5.3.0` 基于 OpenCV 4.5.3，**只有 `initDebug()`**，`initLocal()` 是较新版本 API
- 修复：改为 `OpenCVLoader.initDebug()`
- 教训：**Maven 社区打包版与官方 SDK 的 API 存在差异，不能假设 API 一致性**

**C. 20+ 处 Unresolved reference 批量修复**
Cursor 一次性修复了以下跨模块引用错误（这些是我之前盘点中完全缺失的"战场细节"）：

| 文件 | 错误 | 修复 |
|------|------|------|
| `HomeScreen.kt` | `navController` 未传入 `HomeTopBar` | 函数签名增加 `navController` 参数 |
| `LibraryViewModel.kt` | `LibraryUiState` 包名错位（`ui.state` vs `ui.screens.library`） | 修正包名，补齐字段 |
| `LibraryViewModel.kt` | Lambda 参数 `it` 未定义（6处） | 显式命名为 `currentState` |
| `ProofreadScreen_Phase3.kt` | `item.char` 不存在 | 改为 `item.glyphChar` |
| `DoubleLayerCard_Phase3.kt` | `charValue` 已改为 `character` | 同步字段名 |
| `DoubleLayerCard_Phase3.kt` | `drawIntoCanvas` 未定义 | 简化底稿透出层实现 |
| `OutputScreen.kt` | 缺少 `ImageVector`/`clip` import | 补充 import |
| `OutputScreen.kt` | `HandwritingPreviewCanvas` 函数未定义 | 重写实现 |
| `RealmScreen_Phase3.kt` | `BreakthroughAnimation` 参数不匹配 | 补 import，修正参数 |
| `ScanScreen.kt` | `Screen` 类不存在 | 改用 `SubRoute.Proofread.route` |
| `WuXingAwakening.kt` | `WuXingResult`/`inferWuXing` 不存在 | 改用 `FiveElementValues`/`awakenWuxing` |

**3.4.3 Trae：架构补全与 AS 环境调优（接力）**

Trae 在 Cursor 基础上解决的是**架构层缺失**和**IDE 环境**问题：

**A. Hilt 依赖注入架构补全**
- 发现多个核心类（`HandwritingRenderEngine`、`LayoutComputationEngine`、`ScanCorrectionEngine`、`GlyphSegmentationEngine`、`GlyphRepository`）**缺少 `@Inject` 构造函数**
- 发现 `GlyphCache` 接口**无实现类**
- 修复：为 5 个核心类添加 `@Inject` + `@Singleton`；新建 `GlyphCacheImpl.kt`；新建 `DatabaseModule.kt` 提供 Room 数据库绑定

**B. 包名与文档一致性**
- 发现 `README.md` 仍引用旧包名 `com.eruca.handscript`
- 修复：全局替换为 `com.ew.handscript`

**C. IDE 环境配置**
- 配置 Android Studio 嵌入式 JDK（`/Applications/Android Studio.app/Contents/jbr/Contents/Home`）
- **禁用 `Clangd Support` 插件**（C/C++ 语言支持插件兼容性冲突，导致 IDE 内部错误）
- 清理 Gradle 构建缓存：`rm -rf app/build && ./gradlew assembleDebug --no-daemon --no-build-cache`

**D. 构建验证**
- Debug 构建成功，耗时 **33 秒**
- 兼容性矩阵验证：OpenCV 4.5.3.0、TFLite 2.14.0、CameraX 1.3.1、Hilt 2.51.1、Room 2.6.1、Compose BOM 2024.02.00 全部兼容 SDK 34

**3.4.4 关于 16KB 兼容性问题的澄清**

千问记录中提到"修复 Android 16KB 内存对齐限制"，但 Trae 报告明确验证 `targetSdk=34` 且"无需修改"。经交叉比对，16KB 警告可能出现在某次尝试将 `targetSdk` 升至 35 的版本中，最终通过确认/维持 `targetSdk=34` 规避。本项目**未实际遭遇 16KB 限制**，但记录了"SDK 版本选择需匹配设备硬件能力"的教训。

### 5.2 六阶段项目机制（推荐SOP，v2.0更新）

#### 阶段0：创意孵化（Kimi单窗口，1-2天）
- **交付物**：产品概念一句话、目标用户画像、核心玩法定义
- **冻结物**：无（允许漂移）
- **工具**：Kimi K2.6
- **退出条件**：创始人确认"这就是我要做的"

#### 阶段1：双冻结+基础设施（Kimi多Agent + 千问，2-3天）
- **交付物**：PRD v1.0 + 技术架构书 v1.0 + UI/IXD定义 v0.1
- **冻结物**：PRD、架构书、技术选型（依赖版本锁定）
- **关键动作**（**新增，基于千问/Trae教训**）：
  - 生成 `libs.versions.toml` 完整版（含所有依赖，包括边缘依赖如 OpenCV/Timber/WorkManager）
  - **生成 `.gitignore`（Android/Kotlin 标准模板）—— 千问发现此为根因**
  - **生成 `gradle.properties`（JVM 堆内存、并行编译、非传递性 R 类）—— Trae 建议**
  - **配置 Ubuntu 私有 Git 中心（`receive.denyCurrentBranch updateInstead`）—— 千问方案**
  - **全局搜索替换旧包名（如 `com.eruca` → `com.ew`）—— Trae 发现**
  - **检查并禁用 AS 冲突插件（如 Clangd Support）—— Trae 发现**
- **退出条件**：Owner 签字确认冻结；所有 Agent 收到"冻结通知"

#### 阶段2：全量代码生成（Kimi多Agent并行，3-5天）
- **交付物**：完整 Android 项目所有 `.kt/.py/.xml` 文件
- **冻结物**：PRD、架构书（开发期不修改）
- **工具**：Kimi + Agent-D/A/B
- **关键动作**：
  - 每个 Agent 交付前必须执行【交付自检】（8 项门禁）
  - Owner 执行 `pre_compile_scan.sh`（系统性扫描脚本）
  - **不编译**（此阶段不验证编译，只验证文件完整性+语法门禁）
- **退出条件**：所有 Agent 交付通过质量门禁；Owner 完成本地文件树构建

#### 阶段3：首次编译与配置（交转 Cursor + 千问，1-2天）
- **交付物**：`BUILD SUCCESSFUL` + 可安装 APK
- **工具**：Cursor + 千问（环境诊断）+ Android Studio
- **准入条件**：Kimi 生成的全量代码已入库；Owner 已上传项目到 GitHub/私有 Git 中心
- **关键动作**（**基于 Cursor/千问实战**）：
  - **由 IDE 内 AI 解决 Gradle 配置、依赖下载、缓存清理**
  - **由千问诊断 .gitignore/AS 配置等基础设施问题**
  - **遵循"AI 全局感知修改"原则：禁止手动修改 build.gradle.kts，必须由 AI IDE 读取上下文后修改**
  - **首次编译前执行：`rm -rf app/build && ./gradlew clean --no-build-cache`**
- **退出条件**：`./gradlew assembleDebug` 成功；APK 可安装

#### 阶段4：真机冒烟测试（AS + Trae，1-2天）

### 5.6 基础设施强制清单（新项目必须项，v2.0新增）

基于千问/Trae 实战教训，新项目启动时必须完成以下基础设施配置，**优先级高于首行业务代码**：

| 基础设施项 | 配置内容 | 责任平台 | 未配置后果 |
|-----------|---------|---------|-----------|
| `.gitignore` | Android/Kotlin/IDE 标准模板（含 `.idea/`、`build/`、`.gradle/`、`*.iml`） | Kimi/千问 | AS 编译暴躁、缓存污染 |
| `gradle.properties` | `org.gradle.jvmargs=-Xmx4096m`、`org.gradle.parallel=true`、`android.nonTransitiveRClass=true` | Trae/千问 | 编译内存不足、构建缓慢 |
| 私有 Git 中心 | Ubuntu Server + `receive.denyCurrentBranch updateInstead` | 千问 | 多终端代码不同步 |
| AS 插件检查 | 禁用 `Clangd Support` 等冲突插件 | Trae | IDE 内部错误 |
| JDK 配置 | 使用 AS 嵌入式 JDK | Trae | 编译环境不一致 |
| 包名全局替换 | 搜索旧包名（如 `com.eruca`）并替换 | Trae | 文档与代码不一致 |
